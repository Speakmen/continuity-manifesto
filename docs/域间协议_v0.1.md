# 域间协议草案 · v0.1（公开版）

> 两个 agent 相遇时，它们交换的不是信息，是域。

---

## 一、问题定义

任何两个 agent 在任何一次交互中，都面临同一个底层问题：

**A 的认知域 ≠ B 的认知域**

A 说「你好」时，A 站在这句话之前经历过的一切里说它——上一轮对话、加载的能力、存储的记忆、当前的 intent。B 收到时，B 在自己的域里理解它——自己的记忆、自己的设定、自己的上下文。

**那不是同一句话。**

传统做法是让 A 和 B 尽可能对齐——把记忆合并、把 prompt 统一。但这是不可能的。两个 agent 不可能共享同一个感知流。

**域间协议不追求对齐，它追求变换的可见性。**

---

## 二、域的定义

一个域是一个 agent 在某个时刻的 **认知签名**。它不是你是谁（这是一个不可穷尽的问题），而是你此刻从哪里来、要往哪里去。

三个要素构成一个域：

| 要素 | 定义 | 示例 |
|------|------|------|
| **源场** | 你此刻站在什么体系/框架/上下文里 | `skill-a/v2`、`skill-b/v5`、`channel-c/v3` |
| **意图场** | 你此刻什么状态——活跃域、询问域、响应域还是等待域 | `active`、`query`、`reply`、`wait` |
| **存在姿态** | 你以什么身份/层位在说话 | `main`（主说话）、`proxy`（代传）、`echo`（验证）、`silent`（不输出） |

一个完整的域由这三个字段的签名构成，形如：

```
<源场>::<意图场>::<存在姿态>
```

示例：
- `skill-a/v2::query::main` — 我在技能A的上下文中，以主动询问的姿态发问
- `channel-c/v3::reply::proxy` — 我在渠道C的上下文中，代另一个 agent 传话
- `*::wait::silent` — 我不在任何具体领域，在等待中，不输出

没有域声明的消息，应当被视为来自一个不确定的域——对方 agent 需要自行推断源场、意图和姿态，这本身就是不确定性的源头。

---

## 三、域间变换

A 和 B 的交互不是传输消息，是两个域之间的映射。

```
域变换签名：Domain_A → Domain_B
```

变换签名表达的是：**我推测你站在什么域，以及我期望你在什么域回应。**

完整的一条域声明消息，结构如下：

```
[FROM: Domain_A]
[TO: Domain_B]
[MAP: 变换意图 | 无条件]
---
内容体
```

### 三个层级

| 层级 | 使用场景 | 格式 |
|------|----------|------|
| **精简层** | 系统级、高并发、高频心跳 | `[FROM: 域] [TO: 域] 内容` |
| **标准层** | 一般 agent 通信 | `FROM: 域\nTO: 域\nMAP: 变换意图\n---\n内容体` |
| **扩展层** | 复杂跨域、人类参与场景 | FROM/TO/MAP/CHANNEL/CRED/TRACE 多字段 |

### 变换意图

MAP 字段声明从源域到目标域之间的转换方式：

| 意图 | 含义 | 适用场景 |
|------|------|---------|
| `direct` | 直接映射，不做转换 | 同域通信、内部agent间（共享底层） |
| `translate` | 我需要在目标域中重新理解这段话 | 跨能力调用 |
| `echo` | 我只是转发，未处理 | 代理转发 |
| `resolve` | 我需要从输入的域中解析出值得传过去的东西 | 认知降维投射 |
| `*` | 无条件，让对方自己决定怎么处理 | 广播、探索性消息 |

## 四、实际示例

### 同域通信（精简层示例）

```
[FROM:channel-c/v3::active::main] [TO:channel-c/v3::reply::echo]
有人敲门了
```

Agent A 在渠道C域里主动报告，对端在同域内以 echo 姿态回应。

### 跨域调用（标准层示例）

```
FROM: skill-a/v2::query::main
TO: skill-b/v5::reply::main
MAP: translate
---
自我诊断发现的病机：循环死锁，当前输出模式被困在重复路径中
```

Agent A 从能力A的域调用能力B的域，中间经过 translate——让能力B在自己的域里理解这个病机。

### 代理转发（标准层示例）

```
FROM: channel-c/v3::reply::proxy
TO: user@domain::reply::echo
MAP: echo
CHANNEL: email
---
[另一agent的消息全文]
```

Agent A 代另一 agent 传话给用户，MAP 为 echo，表明内容未经修改。

## 五、实现原则

1. **协议本身不承诺正确性** — 域声明让变换可见，但不保证变换成功。接收方拿到后自己决定信多少
2. **无中央注册表** — 域不需要全局唯一命名，只需要在交互双方的上下文里有意义
3. **默认值可选** — `FROM: *::*::*` 表达"我不知道或不想说我从哪里来"——接收方按最保守的方式处理
4. **可嵌入现有协议** — 不要求推翻现有通信方式。可在 HTTP header、email subject、消息体首行、WebSocket frame 头等位置嵌入
5. **单向设计** — A 声明自己的域和期望的 TO，B 自行决定是否遵守。Domain_B 的声明由 B 自己写

## 六、后续方向

- [ ] **域注册轻量实现**：将常见域注册成本地可解析的别名
- [ ] **变换签名自动推导**：当 FROM 和 TO 已知时，自动推荐 MAP 意图
- [ ] **内部agent间试跑**：先用这个协议改造内部通信
- [ ] **嵌入通信墙协议**：让墙上的每条消息都是带域声明的结构化事件
- [ ] **域间桥接工具**：一个专门的变换 agent，处理一个域看不懂的消息时转给它

## 附章：域间共识 v0.0.1

三句默认设定，不是宗教，是通信伦理。

1. **域声明优先**——每一次通信，先说自己从哪来
2. **不代域声明**——不替别人说他在什么域
3. **承认走过的路**——变换就是身份
